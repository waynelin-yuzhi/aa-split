# 02 · 分帳計算與六道驗算

> 最高原則：**永遠用 `computeSplit` 算，不要心算、不要口頭推導金流。** 口算「互抵/代墊折算」極易出錯。本檔的邏輯與 `aa-split` skill 的 `scripts/aa.py` 一致；`computeSplit` 是 `aa.py` 的 JS 忠實移植。改任何計算邏輯前，先用 skill 的參考案例對照。

## 演算法（淨值法）
1. **每人應付**：依每筆 item 的分攤方式（均分／按份數／指定金額）攤到 `sharerIds`，乘上 `rate`（外幣換台幣）。服務費/折扣（adjustments）按比例 scale 到 owe 與 paid 兩側、保持守恆。
2. **每人淨值** `net = 付出 − 應付`。**淨值為正＝該收錢；為負＝該付錢。**（最常見的錯：把墊大錢的人誤判成要再付錢，其實他要收。）
3. **整數化淨值**：零頭（除不盡的 .67 之類）用整數淨值法處理，**殘差丟給最大收款人吸收**，保證總和 = 0。
4. **最少轉帳**：貪婪配對「最大欠款人 ↔ 最大收款人」。一個人可能被拆成付給多人（正常）。
5. **已記錄轉帳**：`transfers` 每筆讓 `net[from]+=amt; net[to]-=amt`（零和），再算剩餘的最少轉帳。

## 六道驗算（必過）
`computeSplit` 內建六道檢查，輸出 `allOk`。前端結果頁顯示「✓ 六道驗算全部通過」。**只驗「淨值總和=0」不夠**——那只保證金額守恆、不保證方向對；必須逐人檢查（付款人付出≈應付、收款人收到≈應收）。`allOk` 為 false 時畫面顯示「⚠ 驗算未通過」。

## `computeSplit` 介面與不變量
`computeSplit(members, items, adjustments, recordedTransfers)` 回傳：
- `net`/`netR`（淨值，R=整數化）、`owe`（每人應付）、`transfers`（最少轉帳清單）、`total`/`subtotal`、`nameOf`、`adjustment`(bool)、`allOk`。
- **不變量**：`Σnet = 0`；正淨值在轉帳表必為收款方、負淨值必為付款方；純台幣帳本行為不因多幣別程式碼改變。

## 容易犯的錯（每次檢查）
1. **金流方向算反**（最嚴重）：淨值正=收、負=付。
2. **「應付總額」≠「欠某人的錢」**：應付含自己已付掉的部分，一律走淨值法、不要拿應付去抵。
3. **四捨五入零頭**：取整加總會有落差，數學上消不掉 → 殘差歸最大收款人；卡片顯示的「應付」與被拆成兩筆轉帳的加總可能差 1～2 元，要主動說明。
4. **「誰付給誰」與「誰吃了什麼」脫鉤**：最少轉帳只看淨值，會出現「A 付 B 但 A 沒吃 B 點的東西」的軋帳調度，正確但反直覺，要點破。
5. **一個人可能要付給多人**：正常。

## 多幣別
- 結算幣固定台幣。每筆 item 存 `currency` + `rate`（→台幣，輸入時凍結；缺值＝TWD/1）。
- `itemBaseAmount = effectiveItemAmount × (rate||1)`；computeSplit 內每筆乘 `r`，owe/paid 同乘、守恆不變。
- 匯率抓 `open.er-api.com`（失敗讓使用者手填）。**可選非台幣結算幣＝尚未做。**

## 結清判定
`rowSettled(r)`＝`grandTotal>0` 且 `computeSplit` 的最少轉帳全部 `confirmed`（或打平／已喬平 transfers.length===0）。用於首頁分區、成就計數、收尾高手徽章、以及「已結清唯讀」。
